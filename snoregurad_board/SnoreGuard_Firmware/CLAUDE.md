## Architecture

### FreeRTOS Task Map

| Task | Priority | Stack (words) | Created in |
|------|----------|---------------|------------|
| `audio_inference_task` | 4 | 4096 | `source/main.c` |
| `button_task` | 5 (highest user task) | 2048 | `source/app_hw/app_hw_device.c` via `app_bt_hw_init()` |
| `level_btn_task` *(optional)* | 4 | 512 | `source/app_hw/app_hw_device.c` — only when `SNOREGUARD_ENABLE_LEVEL_BUTTON` is defined |
| FreeRTOS timer service | 3 | 256 | FreeRTOS internals |
| BT stack internals | varies | managed by WICED | `wiced_bt_stack_init()` |

**Critical scheduling notes:**
- `button_task` must be higher priority than `audio_inference_task` so GPIO interrupts are serviced immediately.
- `audio_inference_task` uses `vTaskDelay(pdMS_TO_TICKS(1))` (not `taskYIELD()`) at the end of each buffer cycle. `taskYIELD()` only yields to equal-or-higher priority tasks and would starve button_task; `vTaskDelay` puts the task in BLOCKED state and allows any lower-priority task to run.
- `button_task` stack is 2048 words (8 KB). It calls into `wiced_bt_gatt_server_send_notification` for morning sync, which uses significant BT stack depth. Reducing below 2048 causes a silent stack-overflow crash.
- `vApplicationStackOverflowHook` is defined in `source/main.c` and prints `!!! STACK OVERFLOW in task: <name> !!!` to UART before asserting.

### Data Flow

```
PDM/PCM mic (16 kHz)
  → IMAI_enqueue(sample)           [models/model.c – Imagimob generated]
  → IMAI_dequeue(label_scores)
  → audio hysteresis (0.35 entry / 0.20 retain)    [source/main.c]
  → snore_detect_on_decision(is_snore)              [source/snore/snore_detect.c]
      → process_snore_frame()
          → posture_validation_on_frame()           [source/snore/posture_validation.c]
              → (on fail) snore_haptic_level_increment()
          → sliding window check
          → trigger_haptic()
              → haptic_motor_fire()                 [source/app_hw/app_hw_device.c]
                  → 6-round shake sequence via FreeRTOS timer
                  → snore_on_haptic_complete()      [source/snore/snore_detect.c]
              → posture_validation_start()          [source/snore/posture_validation.c]
          → snore_log_add_event()                   [source/snore/snore_flash_log.c]
              → SRAM ring buffer (s_buf[200])
              → snore_log_flush_to_flash() → mtb_kvstore_write(&kvstore_obj, ...)
```

### Timestamp Strategy (Priority Order)

Managed across `source/snore/snore_detect.c` and `source/snore/rtc_time.c`:

1. **Hardware RTC** (`rtc_time_is_valid() == true`) — accurate calendar time after BLE Time Sync seeds it via `rtc_time_set_unix(epoch)`. Survives BLE disconnects.
2. **BLE epoch + uptime** (`TIME_SYNC_OK`, RTC not yet seeded) — `epoch_base + (uptime - uptime_at_sync)`
3. **Uptime fallback** (`TIME_SYNC_FALLBACK` / `TIME_SYNC_STANDALONE`) — `1700000000 + uptime`

States in `time_sync_state_t`:
- `TIME_SYNC_OK`: phone wrote epoch → both fallback math and RTC are seeded
- `TIME_SYNC_FALLBACK`: connected but no write within 30 s → uptime epoch
- `TIME_SYNC_STANDALONE`: no BLE connection within 5 min → uptime epoch

Both timeout timers are FreeRTOS software timers created in `app_bt_application_init()` (`source/app_bt/app_bt_event_handler.c`).

### BLE Service (Sleep Monitor, 128-bit UUID base `E0-1F-56-98...`)

Handles are defined in `GeneratedSource/cycfg_gatt_db.h`. Write logic lives in `app_bt_set_value()` in `source/app_bt/app_bt_gatt_handler.c`.

| Handle | Characteristic | Direction | Action on write |
|--------|---------------|-----------|-----------------|
| `0x000C` | Time Sync | Phone → Device | `snore_set_epoch_base(epoch)` + `rtc_time_set_unix(epoch)` + stops 30-s timer |
| `0x000E` | Log Transfer | Device → Phone (Notify) | Streams 7-byte event packets on morning sync |
| `0x000F` | Log Transfer CCCD | Phone → Device | Saves to kv-store via `app_bt_update_cccd()` |
| `0x0011` | Haptic Intensity | Phone → Device | `snore_set_haptic_level(0-4)` |
| `0x0013` | Sync Ack | Phone → Device | `0x01` → `snore_log_clear()`; `0x00` → log preserved |
| `0x0015` | Haptic Enable | Phone → Device | `0x01` → `snore_set_haptic_enabled(true)`; `0x00` → motor suppressed |

**7-byte notification packet layout** (little-endian):
```
[0..3] uint32  timestamp (Unix epoch or fallback)
[4]    uint8   duration_s
[5]    uint8   haptic_success
[6]    uint8   haptic_flag
```

Morning sync is triggered by a short button press (50–250 ms) → `app_bt_morning_sync_start()` in `source/app_bt/app_bt_gatt_handler.c` → sends first notification → each `GATT_APP_BUFFER_TRANSMITTED_EVT` triggers the next via `app_bt_morning_sync_send_next()` → after last event, sets `s_pending_ack = true` (log is **not** cleared yet) → app writes `0x01` to Sync Ack (`0x0013`) after SQLite insert → `snore_log_clear()` is called only then. If the app disconnects or fails before writing ack, the log is preserved and re-transmitted on the next sync.

### Flash Storage (kv-store)

**Two variables — do not confuse them:**
- `mtb_kvstore_bd_t block_device` — the raw block-device interface (defined in `source/app_bt/app_bt_bonding.c`). Passed to `app_kvstore_bd_config()` and `mtb_kvstore_init()`. Never passed to read/write/delete.
- `mtb_kvstore_t kvstore_obj` — the kvstore handle (defined in `source/app_bt/app_bt_bonding.c`, declared `extern` in `app_bt_bonding.h`). This is the first argument to every `mtb_kvstore_read/write/delete` call.

Two consumers share `kvstore_obj`:
1. **BLE bonding** (`source/app_bt/app_bt_bonding.c`) – link keys, CCCD, identity keys
2. **Snore log** (`source/snore/snore_flash_log.c`) – keys `sg_count`, `sg_data`

**Storage partition:** `AUXILIARY_FLASH_LENGTH = 64` in `source/app_hw/app_flash_common.h` → 64 × 512-byte pages = 32 KB at the end of internal flash Block 1. The original 16-page (8 KB) allocation caused `MTB_KVSTORE_STORAGE_FULL_ERROR` (error code `0x09420006`) under rapid writes.

**Flash log format:** `flash_save()` linearizes the circular buffer and writes only the `s_count` active events (`s_count × 7` bytes) rather than the full 1400-byte array. On `flash_load()`, events are read linearly into `s_buf[0..s_count-1]` and `s_head = s_count % SNORE_LOG_MAX_EVENTS`. The `sg_head` key is no longer stored — head is always derived from count after a linearized load.

### Haptic Motor

`cyhal_pwm_t` on `HAPTIC_MOTOR_PIN` (default `CYBSP_USER_LED1` for dev-kit visual feedback; change to actual motor GPIO before hardware deployment). 5 duty-cycle levels (20–100%) in `HAPTIC_DUTY_PERCENT[]`.

**Shake Sequence (merged from `code/Motor_Controll.c`):**
`haptic_motor_fire()` now performs 6 consecutive on/off pulses (500 ms on / 500 ms off) driven by a one-shot FreeRTOS timer (`haptic_shake_timer`) re-armed in its own callback. State machine: `SHAKE_ON → SHAKE_OFF → SHAKE_ON ...` for `HAPTIC_SHAKE_ROUNDS` iterations, then calls `snore_on_haptic_complete()`.

The old single 3-second burst (`haptic_stop_timer`) has been replaced by this pulse sequence.

### AI Model

`models/model.h/c` is the Imagimob speech demo model (3 classes at runtime per the current build, `IMAI_DATA_OUT_COUNT` reported as 3). The real snore model replaces these files. In `source/main.c`, `SNORE_LABEL_IDX=-1` means auto-detect (any non-`unlabelled` class with score ≥ 0.5 counts as a snore). Set it to the correct class index after swapping the model.

**Audio Label Hysteresis (merged from `code/Audio.c`):**
The audio task applies stickiness before calling `snore_detect_on_decision()`:
- A new non-silent label is accepted only if its score ≥ `AUDIO_LABEL_ENTRY_THRESHOLD` (0.35).
- Once active, the label stays sticky as long as its score ≥ `AUDIO_LABEL_RETAIN_THRESHOLD` (0.20).
- Below both thresholds → reverts to silent. This reduces rapid flicker without raising the detection threshold.

### Episode Merging (debounce)

**Before:** Every breath was one event. A snore cycle is roughly:

```
inhale (snore sound) → exhale (silent) → inhale (snore sound) → exhale ...
  3 sec                  1 sec             3 sec                  1 sec
```

The firmware watched for the falling edge (snore sound stops) and wrote a new event each time. A single 5-minute continuous snoring stretch produced ~75 events; a full night could exceed 200 and overflow the ring buffer.

**After:** The unit of logging is now a **snore episode**, not a breath. An episode opens on the first rising edge and stays open across successive breaths. It closes only after `SNORE_EPISODE_GAP_S` (15 s) of continuous silence. At close time, a single `snore_event_t` is logged with `timestamp` = episode-start (wall-clock) and `duration_s` = start → last-falling-edge (clamped at `SNORE_EPISODE_MAX_DUR_S` = 240 s, since the field is `uint8_t`). Episodes shorter than `SNORE_EPISODE_MIN_DUR_S` (3 s) are discarded as noise.

**State machine inside `process_snore_frame()`:**

| Event | Action |
|-------|--------|
| Rising edge (snore starts), no episode open | Open episode: record `s_episode_start_ts` (wall-clock, goes in packet) and `s_episode_start_up` (uptime, used for duration math). |
| Rising edge, episode already open | No-op — just another breath inside the same episode. |
| Falling edge (snore stops) | Update `s_episode_last_snore_up = now`. **Do not log.** Still evaluate the sliding-window haptic trigger. |
| Every frame, silent, episode open, `now - s_episode_last_snore_up ≥ 15 s` | Close episode. If `duration_s ≥ 3` → log one event. Else discard. |

**Constants and rationale:**

| Constant | Value | Reason |
|---|---|---|
| `SNORE_EPISODE_GAP_S` | 15 s | Long enough that inter-breath pauses (~1–2 s) don't split episodes; short enough that rolling over or coughing counts as a real separator. |
| `SNORE_EPISODE_MIN_DUR_S` | 3 s | Filters single-frame model misclassifications. |
| `SNORE_EPISODE_MAX_DUR_S` | 240 s | `duration_s` is `uint8_t` (max 255). Runs longer than 240 s are capped; the next rising edge opens a fresh episode. |

**Worked example:** 20 minutes of snoring → 30 s silence (rolling over) → 5 minutes of snoring.
- Old: ~400 events.
- New: 2 events — first with `duration_s = 240` (capped), second ditto. The 30 s silence exceeds the 15 s gap so the first episode closes. If the silence had been only 10 s, the two runs would have merged into one episode.

**Haptic interaction:** The 5-breaths-in-60-s sliding window still fires on each qualifying falling edge. When it does, the current episode is handed off to the existing 15-min post-haptic evaluation path (`s_pending_haptic_event`) with `haptic_flag = 1`, and `s_episode_open` is set to `false` so the gap-close tick does not double-log the same event. `haptic_success` is filled in later by `evaluate_post_haptic()` as before.

**Morning sync:** `app_bt_morning_sync_start()` calls `snore_detect_flush_open_episode()` before streaming so that if the user presses the button while still snoring, the open episode is closed and logged in place rather than lost.

**Net effect:** nightly event count drops from ~200–400 to ~20–40. Flutter app timeline shows fewer, longer dots representing real episodes instead of individual breaths. Wire format and SQLite schema on the app side are unchanged.

### Posture Validation (merged from `code/Detection_Logic.c`)

After the haptic motor fires, `posture_validation.c` opens a 10-second window:
- **SUCCESS**: 10 consecutive seconds of no-snore → no action (level unchanged).
- **FAIL**: snore resumes before the window closes → `snore_haptic_level_increment()` bumps the haptic level by 1 (clamped at 4).

This is an **additional layer** on top of the existing 15-minute post-haptic window in `snore_detect.c`. Both run concurrently: the 10-s window drives level escalation, the 15-min window drives the `haptic_success` field in the flash log.

### Hardware RTC (merged from `code/RTC.c`)

`source/snore/rtc_time.c` wraps `cyhal_rtc_t` and is initialised in `main()` before the BT stack. Once the phone writes the Time Sync characteristic, `snore_set_epoch_base()` calls `rtc_time_set_unix()` to seed the RTC. From that point, `snore_get_timestamp()` returns `rtc_time_get_unix()` — accurate calendar time that survives BLE disconnects.

## Companion Android App (2026-04-16)

A complete Flutter Android companion app was implemented at `D:\ACADEMIC\snoreguard_project\snoreguard_app\` and built to a release APK at `D:\ACADEMIC\snoreguard_project\snoreguard_app.apk`.

### BLE Protocol Compatibility

The app was built to match this firmware's GATT service exactly. UUIDs were derived from `GeneratedSource/cycfg_gatt_db.h`:

| Characteristic | UUID | App usage |
|---|---|---|
| Service | `e01f5698-4b21-4710-a0f6-001122334455` | Service discovery filter |
| Time Sync | `e11f5698-4b21-4710-a0f6-001122334455` | Write 4-byte uint32 LE epoch on connect |
| Log Transfer | `e21f5698-4b21-4710-a0f6-001122334455` | Subscribe notify, parse 7-byte LE packets |
| Haptic Intensity | `e31f5698-4b21-4710-a0f6-001122334455` | Read/Write 1-byte value 0–4 |
| Sync Ack | `e41f5698-4b21-4710-a0f6-001122334455` | Write `0x01` after SQLite insert succeeds; triggers log clear |

Device advertised name: `"SnoreGuard"` (matched in scan filter).

**Advertising packet fix (2026-04-17):** The original `cycfg_gap.c` (from the Hello Sensor template) advertised the short name `"SG"` and a mismatched 128-bit UUID. The Flutter app's `withNames: ['SnoreGuard']` filter never matched, so the device was invisible to the app (though nRF Connect could find it). Fix: changed the advertised name to the full `"SnoreGuard"` (10 bytes) and removed the 128-bit UUID from the advertising packet to stay within the 31-byte BLE advertising payload limit. The service UUID is still discoverable via GATT service discovery after connection.

### 7-Byte Packet Decoding

App validates and rejects: packets not exactly 7 bytes, `haptic_success > 1`, `haptic_flag > 1`, zero-duration events with no haptic flag. All others are decoded and stored.

### End-of-Stream Detection

The firmware sends no explicit EOS marker after morning sync. The firmware requires a **physical button press** to start streaming events (see `app_bt_morning_sync_start()` in `app_bt_gatt_handler.c`). The app uses a two-phase timeout: a **60-second initial wait** for the user to press the button, then a **2-second silence timeout** after the first notification to detect end-of-stream. This was fixed on 2026-04-17 — the original implementation started the 2-second EOS timer immediately on CCCD enable, which expired before the user could press the button.

### Timestamp Handling

- **Valid timestamps** (`>= 1731536000`): decoded as real calendar time; session date computed with 6 AM boundary (events before 06:00 local belong to the previous calendar day).
- **Fallback timestamps** (`>= 1700000000 && < 1731536000`, i.e. ~Nov 2023): flagged as `isFallbackTimestamp = true`, grouped under today's date, and shown with a warning in the UI. This matches the firmware's `1700000000 + uptime_s` fallback.

### Haptic Auto-Escalation Awareness

After morning sync, the app reads back the Haptic Intensity characteristic. This accounts for firmware auto-escalation via `snore_haptic_level_increment()` (posture validation fail path) during sleep — the app UI always reflects the current device level, not just what was last written by the user.

### App Architecture Summary

- **State management:** Provider pattern (`BleProvider`, `SessionProvider`, `SettingsProvider`)
- **Storage:** SQLite via sqflite; UNIQUE index on `(event_timestamp, duration_s, haptic_flag)` prevents double-sync duplicates via `INSERT OR IGNORE`
- **Screens:** Onboarding (3-page, first-launch), Home (7-day history + weekly summary bar chart), Session Detail (scatter timeline + stats), Settings (scan/connect, morning sync, haptic slider)
- **Theme:** Dark (sleep-friendly), background `#121220`, primary `#6C63FF`
- **Min SDK:** Android API 29 (Android 10) — required for `BLUETOOTH_SCAN` without location
