Abstract

This project presents a hardware prototype built on the Infineon PSoC™ 6 CY8CKIT-062S2-AI development board, exploring the feasibility of deploying Edge AI for real-time snore detection in a resource-constrained embedded environment. A Conv1D Neural Network optimized via 8-bit Integer Post-Training Quantization (PTQ) is benchmarked across 8 audio preprocessing pipelines to identify the most suitable on-device inference configuration. When repeated snoring is detected, a vibration motor provides haptic feedback through a Sliding Window intervention algorithm. A companion Flutter Android application communicates via Bluetooth Low Energy (BLE) to retrieve session logs and display a 7-day snore history dashboard, operating fully offline with local SQLite storage. The primary research contribution is a systematic evaluation of Edge AI pipeline trade-offs accuracy, latency, and memory footprint on a real embedded platform, rather than a clinical-grade diagnostic device.

Keywords: Edge AI, Conv1D, PSoC 6, Snore Detection, BLE, Flutter, SQLite, Post-Training Quantization, Positional Therapy

1. Introduction

1.1 Background & Motivation

Snoring is one of the most consistent acoustic indicators of disrupted sleep and is associated with conditions such as Obstructive Sleep Apnea (OSA). Deploying machine learning models capable of detecting snoring patterns directly on low-power embedded hardware without cloud connectivity or expensive sensors represents an active and technically challenging area of Edge AI research [1, 2]. This project is motivated by the opportunity to build a working prototype that explores this challenge hands-on: using the Infineon PSoC™ 6 CY8CKIT-062S2-AI development board as the target platform, the project investigates how to train, quantize, and deploy a Conv1D neural network for real-time snore classification under strict memory and latency constraints. The prototype is intended as a research and learning vehicle for understanding Edge AI deployment trade-offs

And to be a prototype device for future development.

1.2 Problem Statement

The technical challenge in automated snore detection lies in the high complexity of domestic acoustic environments. Background noises such as air conditioning or fans share frequency harmonics with human snoring [3], making discrimination difficult. While deep learning models offer high accuracy, deploying them on battery-powered microcontrollers requires balancing model complexity against latency and power consumption. Traditional cloud-based solutions raise privacy concerns and introduce latency [2]. Furthermore, existing consumer sleep trackers lack active intervention mechanisms and depend on persistent wireless connectivity, reducing battery life and increasing electromagnetic exposure during sleep.

1.3 Objectives

  1. To develop and benchmark 9 audio preprocessing pipelines within the Imagimob DeepCraft environment and identify the optimal Conv1D snore detection model using a Weighted Performance Index (WPI).

  2. To implement a real-time Edge AI inference pipeline on the Infineon PSoC™ 6 Cortex-M4 core using 8-bit Integer PTQ, achieving <200 ms detection-to-intervention latency.

  3. To validate the deployed system under real-world sleep conditions through an overnight pilot test, comparing on-device detection results against ground-truth audio recordings.

  4. To design a firmware-level “Sliding Window” haptic intervention system (5 snore events / 60 s threshold, 10-minute cooldown) using vibration for passive posture correction.

  5. To implement a user-defined haptic intensity control system with five adjustable levels (20%, 40%, 60%, 80%, and 100%) and a post-intervention validation logic to track success/unsuccessful flags based on a 15-minute post-vibration snoring silence threshold.

  6. To develop an Android mobile application featuring a 7-day snore history dashboard, per-session snore event timeline with haptic trigger markers, intervention success/fail indicators, and device settings screen, operating fully offline without any internet connectivity.

2. Literature Review

2.1 Obstructive Sleep Apnea and Positional Therapy

OSA is a prevalent sleep disorder caused by repetitive upper airway collapse during sleep. Snoring is its most consistent clinical indicator. Positional Therapy, which encourages lateral sleep positions, has been clinically validated as an effective non-invasive treatment, capable of significantly reducing snore indices [4]. Vibrotactile stimulation. Haptic cues delivered without fully waking the user has been demonstrated to prompt natural posture adjustment and improve patient compliance compared to CPAP machines [5].

2.2 Edge AI and Embedded Machine Learning

Edge AI deploys machine learning models directly on hardware, providing advantages in latency, privacy, and power efficiency. Post-Training Quantization (PTQ) converts floating-point weights to 8-bit integers, reducing model size and inference time while maintaining acceptable accuracy [6]. The Imagimob DeepCraft toolchain supports the full pipeline from preprocessing design and model training to C-code export and deployment on ARM Cortex-M processors [2].

2.3 Audio Signal Processing for Snore Classification

Mel-Filterbank scaling [7] and Short-Time Fourier Transform (STFT) with temporal windowing [8] are standard techniques for transforming raw audio into perceptually relevant frequency features. Conv1D neural networks are well-suited for sequential audio classification, offering favorable accuracy-to-computation ratios on resource-constrained embedded systems.

2.4 BLE for IoT and Flutter Mobile Development

Bluetooth Low Energy (BLE) is the de-facto standard for short-range, power-efficient communication in wearable and IoT devices. Its GATT (Generic Attribute Profile) architecture using Services and Characteristics. enables structured data exchange between embedded devices and smartphones. Flutter, Google’s cross-platform UI framework, combined with the sqflite package for SQLite access, provides a robust foundation for offline-first mobile health applications with native Android performance [9].

2.5 Vibration Motor and Haptic Feedback in Wearable Systems

Vibration motors, particularly Eccentric Rotating Mass (ERM) and Linear Resonant Actuator (LRA) types, are widely used in wearable and IoT devices to deliver non-auditory haptic feedback. ERM motors operate by spinning an off-center mass to generate oscillatory vibration, offering low cost and simple PWM-based control, making them well-suited for microcontroller-driven embedded systems (Immersion Corporation, 2014). Vibrotactile stimulation the delivery of mechanical vibration to the skin has been studied extensively as a means of prompting user response without visual or auditory disturbance, which is particularly relevant for sleep-related applications where preserving sleep continuity is critical (Jones & Sarter, 2008).

In the context of positional therapy for snoring and OSA, vibrotactile feedback has been validated as an effective passive intervention. Studies have shown that gentle vibration applied at the appropriate stimulus threshold can prompt a sleeping user to shift body position without causing full arousal, thereby reducing snoring frequency while maintaining sleep quality [5]. PWM duty cycle control provides a straightforward mechanism for adjusting vibration intensity on embedded platforms, allowing intensity levels to be programmatically defined and modified without hardware changes. This project adopts an ERM vibration motor controlled via PWM from the PSoC™ 6 GPIO, with five discrete intensity levels (20–100%) mapped directly to PWM duty cycle values, enabling user-adjustable haptic feedback through the companion mobile application.

2.6 Infineon PSoC™ 6 Microcontroller

The PSoC™ 62S2 features a dual-core ARM® architecture: a Cortex®-M4 for high-performance AI inference and a Cortex®-M0+ for low-power system management. It integrates dual XENSIV™ MEMS microphones via PDM-PCM, a CYW43439-based BLE 5.0 module, and approximately 2 MB internal flash for on-device data persistence. The platform is natively supported by the Imagimob toolchain and Infineon ModusToolbox™ IDE [10].

3. Methodology

3.1 Project Scope

  1. Benchmark 9 audio preprocessing pipelines in Imagimob DeepCraft to identify the optimal feature extraction method for on-device snore detection.

  2. Apply 8-bit Integer Post-Training Quantization (PTQ) to the winning Conv1D model and export as C-code for the ARM Cortex-M4 core.

  3. Implement firmware on the PSoC™ 6 dual-core architecture, covering real-time audio sampling, DSP inference, Sliding Window haptic logic, and Non-Volatile Flash Logging.

  4. Develop an Android mobile application with a 7-day snore history dashboard, per-session snore event timeline, haptic trigger markers, and intervention success/fail indicators, operating fully offline.

  5. Conduct Controlled Acoustic Bench Testing and evaluate system performance against defined KPIs.

3.3 Proposed Approach

Phase 1: AI Model Optimization and Benchmarking

  1. Dataset Preparation: Curate and label around 188:12 minutes of audio at 16 kHz Mono .wav format ( 55:33 minutes : 83:06 minutes snore to Noise distribution. Noise sound such as fans, AC, ambient bedroom sounds, remainder 49:32 minutes remains as unlabeled ambient).

  2. Preprocessing Pipeline Construction: Configure 8 distinct preprocessing combinations in DeepCraft using a common foundation (Raw audio → Sliding Window 1024 → Hann Smoothing → R-DFT & Frobenius Norm) with variable Mel-Filterbank scaling and post-processing steps.

  3. Model Benchmarking: Evaluate all 9 pipelines using a multi-dimensional Weighted Performance Index (WPI) as described in Section 3.4. Pipeline 9 (Imagimob Speech) is included as a performance ceiling reference and evaluated separately from the primary WPI ranking.

  4. Post-Training Quantization: Convert the Engineering Winner model to INT8 format via the Imagimob toolchain and export as optimized C-code for ModusToolbox™ integration.

Phase 2: Firmware and Hardware Integration

  1. Deploy the quantized model on the ARM® Cortex®-M4 core with real-time PDM-PCM microphone sampling.

  2. Implement Adaptive Haptic Logic: Trigger vibration at the user-selected intensity (20-100%). The system then enters a 15-minute monitoring window. If no snoring is detected during this window, a 'Success' flag is assigned; otherwise, an 'Unsuccessful' flag is recorded.

  3. Implement BLE GATT server: Time Sync characteristic (receive epoch timestamp from mobile app at session start) and Log Transfer characteristic (stream binary sleep log on user-triggered Morning Sync).

  4. Dual-Tier Data Persistence: Implement an SRAM-based circular buffer to manage high-frequency logging for up to 200 recent events, complemented by asynchronous non-volatile flash logging to ensure data integrity across power cycles..

Phase 3: Flutter Mobile Application Development

  1. Implement BLE GATT client (flutter_blue_plus): scan, pair, and communicate with the PSoC 6 device.

  2. BLE Time Sync: transmit current Unix epoch timestamp to the device at session start.

  3. Morning Sync: receive and parse 7-byte binary event packets from the device upon user button press; insert parsed records into the local SQLite database.

  4. Dashboard UI: display Home (7-day session list), Session Detail (snore event timeline, haptic trigger timestamps, and post-vibration success/fail indicators), and Device Settings screens.

  5. Auto-purge: delete SQLite records older than 7 days on application launch.

Phase 4: System Validation and Real-World Evaluation

1.  Controlled Acoustic Bench Test: position CY8CKIT-062S2-AI in a controlled acoustic enclosure; play labeled snore/noise audio at 30–60 dB SPL from 0.5–1.0 m distance.

2.  Real-world Pilot Test: Deploy the device in an actual bedroom environment for a minimum of 2 nights. Record ground-truth audio simultaneously using a separate recorder. Compare on-device detection log against manually labeled ground-truth to calculate real-world F1-Score and False Positive Rate.

  3. Evaluate F1-Score, False Positive Rate, Inference Latency, Pre-processing Latency, Peak RAM, and Flash Occupancy across all 8 pipelines.

  4. Validate BLE sync speed, SQLite write performance, and mobile dashboard rendering time against defined KPIs.

3.4 AI Model Benchmarking

3.4.1 Preprocessing Pipeline Configurations

The 9 pipelines form a structured factorial design varying three independent dimensions: (1) amplitude scale type (Power vs. Log), (2) feature compression (with/without DCT), and (3) preprocessing quality (raw vs. HPF+Hann+Scale vs. Imagimob Speech). Pipelines 1–4 apply no pre-filtering or normalization and serve as baselines. Pipelines 5–8 add High Pass Filter, Hann Window, and Min-Max Scale (“Standard Wrapper”) to suppress low-frequency noise and stabilize model input range. Pipeline 9 uses Imagimob’s built-in AI-based Speech engine as a performance ceiling reference. Note that Pipelines 1–4 differ from Pipelines 5–8 in both HPF and Scale simultaneously; the contribution of each variable is discussed in Section 5.

3.4.2 Weighted Performance Index (WPI)

Each of the 9 pipelines is evaluated using a WPI that prioritizes real-world deployment suitability. Pipeline 9 (Imagimob Speech) is treated as a black-box ceiling reference and reported separately; the primary WPI ranking covers Pipelines 1–8:

WPI = (S_(acc) × 0.80) + (S_(res) × 0.20)

S_(acc) = Binary F1-Score = 2 × (Precision × Recall) / (Precision + Recall)

where Precision = TP / (TP + FP), Recall = TP / (TP + FN), treating Snore as Positive and all non-snore classes (Noise, Unlabeled) as Negative

Table 3.2 WPI Scoring Dimensions

  ------------------- --------- ------------------------------------------------------------------------------------------------------------------
  Component           Weight    Metrics

  S_acc (Accuracy)    80%       Binary F1-Score (Precision × Recall trade-off), treating snore as Positive and all non-snore classes as Negative

  S_res (Resources)   20%       Peak RAM Usage (KB) + Flash Occupancy (KB)
  ------------------- --------- ------------------------------------------------------------------------------------------------------------------

The S_acc component adopts a binary classification approach, treating Snore as the Positive class and all non-snore classes (Noise and Unlabeled) as Negative. Precision penalizes False Positives (unnecessary haptic triggers), while Recall penalizes False Negatives (missed snore events), making Binary F1-Score a balanced and deployment-relevant accuracy metric for this application.

Each component score is normalized across Pipelines 1–8 Pipelines 1–8 prior to WPI calculation, where the best-performing pipeline receives a score of 1.0 and all others are scaled proportionally using min-max normalization.

3.4.3 Evaluation Method: Machine-in-the-Loop Testing

Pipeline evaluation is conducted using a Machine-in-the-Loop (MitL) methodology, in which the physical PSoC™ 6 board participates directly in the inference loop rather than relying solely on offline simulation. This approach ensures that all measurement results reflect real on-device behaviour including hardware-specific quantization effects, PDM-PCM signal conditioning, and embedded runtime characteristics.

The evaluation procedure is as follows. First, ground-truth audio is recorded in an actual bedroom environment using the onboard XENSIV™ MEMS microphone (IM72D128V01XTMA1) of the CY8CKIT-062S2-AI development kit, capturing real sleep sessions that contain natural snoring events, ambient background noise, and silence periods. The recordings are labelled at the segment level to form a ground-truth dataset. Second, the labelled audio is injected back into each compiled pipeline firmware build in a controlled replay sequence: the audio buffer is populated with the pre-recorded PCM samples and fed directly to the preprocessing and inference stage of each pipeline model, bypassing only the live PDM capture while keeping all downstream signal processing and neural network execution on-device. Third, the model output label for each audio window is compared against the ground-truth annotation to compute per-pipeline confusion matrices, from which Binary F1-Score (S_(acc)) and peak RAM usage (S_(res)) are derived and used as inputs to the WPI formula defined in Section 3.4.2.

This MitL design preserves ecological validity by using audio captured under real sleep conditions rather than synthetic signals, while maintaining experimental control by replaying identical audio across all 9 pipeline builds. Each pipeline is therefore evaluated under the same acoustic input, making pipeline-to-pipeline comparisons directly comparable and independent of session-to-session variability.

4. Flutter Mobile Application

The Flutter-based Android mobile application serves as the primary user interface of the system. It communicates with the PSoC™ 6 device exclusively via Bluetooth Low Energy (BLE) GATT protocol and stores all data locally using SQLite through the sqflite package. The app retains sleep data for 7 days only records older than 7 days are automatically purged on application launch. No internet connection is required at any point.

4.2 BLE GATT Data Protocol

Communication is structured around two BLE GATT Characteristics within a custom Sleep Monitor Service:

Table 4.2 BLE GATT Characteristics

  ----------------------- -------------- ---------------------------------------------------------------------------------------------------------------------------------------
  Characteristic          Direction      Purpose

  Time Sync (Write)       App → Device   Sends current Unix epoch timestamp (uint32, 4 bytes little-endian) to synchronize the PSoC 6 RTC at session start.

  Log Transfer (Notify)   Device → App   Streams binary sleep log in 120-byte chunks after the user presses Morning Sync. Each chunk contains 20 event records (7 bytes each).
  ----------------------- -------------- ---------------------------------------------------------------------------------------------------------------------------------------

Binary log packet format per snore event (7 bytes total):

Table 4.3 Binary Event Packet Structure (7 bytes per event)

  -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
  Field            Size        Description
  ---------------- ----------- --------------------------------------------------------------------------------------------------------------------------------------------------------------------------
  timestamp        4 bytes     Unix epoch (uint32, little-endian) of event start.

  duration_s       1 byte      Snore event duration in seconds (uint8, 0–255 s).

  haptic_success   1 byte      0x01 if snoring ceased within the 15-minute post-vibration window (Success); 0x00 if snoring resumed (Unsuccessful). Only valid when haptic_flag = 0x01; otherwise 0x00.

  haptic_flag      1 byte      0x01 if a haptic trigger fired within this event; 0x00 otherwise.
  -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------

4.3 SQLite Local Database Schema

All parsed data is stored in a single SQLite table (snore_events) on the mobile device using the sqflite Flutter package. Records older than 7 days are automatically deleted on application launch.

  event_timestamp   INTEGER     NOT NULL      Unix timestamp of the snore event start.

  duration_s        INTEGER     NOT NULL      Snore event duration in seconds (0–255).

  haptic_success    INTEGER     NOT NULL      0x01 = snoring ceased after haptic trigger (Success); 0x00 = snoring resumed or no haptic fired.
